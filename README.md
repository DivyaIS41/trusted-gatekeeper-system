`~# The Trusted Gatekeeper System

## Overview
The Trusted Gatekeeper System is a Zero-Trust secure communication framework designed for IoT–Fog–Cloud environments.  
It ensures that data generated by devices reaches the cloud only after passing through a fog-level security gateway that verifies, inspects, and filters all communication.

## Core Idea
No device or message is trusted by default.  
Every interaction is verified at the fog layer before being forwarded to the cloud.

## Stage A: Basic TCP Communication
This stage creates the first working path:

IoT Device -> Fog Gateway -> Cloud Server

The IoT Device sends messages periodically to the Fog Gateway. The Fog Gateway
receives each message and forwards it to the Cloud Server. The Cloud Server
receives the forwarded message and prints it.

Stage A intentionally has no encryption and no security checks. Those are added
in later stages after the basic pipeline works.

### Files
- `common/network.py`: shared JSON-over-TCP send and receive helpers
- `common/settings.py`: shared host and port settings
- `iot_device/device.py`: creates and periodically sends sample sensor messages
- `fog_gateway/gateway.py`: receives IoT messages and forwards them to Cloud
- `cloud_server/server.py`: receives forwarded messages from Fog

## Stage B: Encrypted Communication
Stage B keeps the same TCP path, but the application message is encrypted with
Fernet before it travels across each network hop.

IoT Device -> encrypted with IoT-to-Fog key -> Fog Gateway

Fog Gateway -> decrypted, wrapped, encrypted with Fog-to-Cloud key -> Cloud Server

Cloud Server -> decrypted and printed

### How Fernet encryption works
Fernet is symmetric encryption from the `cryptography` Python library. Symmetric
means the same secret key encrypts and decrypts a message. In this project stage,
the keys are fixed in `crypto/fernet_crypto.py` so the programs can be restarted
without losing the ability to decrypt messages.

Fernet turns readable JSON like this:

```json
{"device_id": "iot-device-001", "temperature_celsius": 28.4}
```

into an unreadable token string before it is sent over the socket. The receiver
uses the matching key to turn the token back into JSON.

### Where encryption happens
- `iot_device/device.py`: encrypts the sensor message using `IOT_TO_FOG_KEY`
  before sending it to Fog.
- `fog_gateway/gateway.py`: decrypts the IoT message using `IOT_TO_FOG_KEY`,
  then re-encrypts the forwarded message using `FOG_TO_CLOUD_KEY`.
- `cloud_server/server.py`: decrypts the Fog message using `FOG_TO_CLOUD_KEY`
  and prints the plaintext data.
- `crypto/fernet_crypto.py`: contains the fixed keys and shared encrypt/decrypt
  helper functions.

## Stage C: Device Identity and Zero Trust
Stage C keeps the encrypted Stage B pipeline and adds Fog-side trust decisions.
The Fog Gateway now decrypts the IoT message, checks it, and forwards it only if
all Zero-Trust rules pass.

Each IoT message includes:

- `device_id`: the unique identity of the sending device
- `timestamp`: the Unix time when the message was created
- `nonce`: a random one-time value
- `payload`: the sensor data

### Zero-Trust rules at Fog
- Reject the message if `device_id` is not in the trusted device list.
- Reject the message if `timestamp` is older than 30 seconds.
- Reject the message if `nonce` has already been used by that device.

The trusted device list and nonce memory are in `fog_gateway/security.py`.

### How replay attacks work
A replay attack happens when an attacker captures a valid message and sends it
again later. Even if the attacker cannot read the encrypted payload, they may try
to reuse the same encrypted message to trick the system into accepting duplicate
sensor data or duplicate commands.

Stage C reduces this risk in two ways:

- Timestamp check: old captured messages expire quickly.
- Nonce check: a message with the same one-time nonce is accepted only once.

### Why Zero-Trust is needed
Encryption protects message contents while they travel over the network, but it
does not answer every security question. The Fog still needs to ask: Is this a
known device? Is this message fresh? Have I already seen this message?

Zero-Trust means the Fog does not trust a message just because it arrived over a
socket or decrypted successfully. Every message is checked before it reaches the
Cloud.

### Stage C data flow
IoT Device -> add device ID, timestamp, nonce -> encrypt with IoT-to-Fog key

Fog Gateway -> decrypt -> check trusted device -> check timestamp -> check nonce

Fog Gateway -> re-encrypt accepted message with Fog-to-Cloud key -> Cloud Server

Cloud Server -> decrypt and print

## Stage D: Attack Simulation and Detection
Stage D adds attack clients in the `attacks` module and detection logic in the
Fog Gateway.

The Fog Gateway blocks and logs:

- Fake device attack: a message uses an unknown `device_id`.
- Replay attack: a message reuses an already-seen `nonce` or arrives with an old
  timestamp.
- Flooding attack: a trusted device sends too many messages in a short time.

Fog logs are written to:

```text
logs/fog_security.log
```

### How to run attack simulations
Start Cloud and Fog first:

```bash
python -m cloud_server.server
python -m fog_gateway.gateway
```

Then run one attack from another terminal:

```bash
python -m attacks.fake_device_attack
python -m attacks.replay_attack
python -m attacks.flooding_attack
```

### Expected attack responses
Fake device:

```text
{'status': 'rejected', 'details': 'Unknown device ID: iot-device-999', 'event_type': 'fake_device_attack'}
```

Replay:

```text
{'status': 'rejected', 'details': 'Nonce was already used.', 'event_type': 'replay_attack'}
```

Flooding:

```text
{'status': 'rejected', 'details': 'Flooding detected: too many messages in a short time.', 'event_type': 'flooding_attack'}
```

## Stage E: Logging and Reporting
Stage E adds file-based logs and a summary report.

Fog writes suspicious and blocked traffic to:

```text
logs/fog_security.log
```

Cloud writes accepted data and blocked-message reports to:

```text
logs/cloud_audit.log
```

Print a summary report after running traffic or attacks:

```bash
python -m reports.summary_report
```

Logs help security analysis because they show what was accepted, what was
blocked, why it was blocked, and when the decision happened.

Dashboard sections:

- System Status: NORMAL or ALERT based on recent suspicious or blocked events.
- Live Data Flow: IoT -> Fog -> Cloud flow with accepted, suspicious, and
  blocked counts.
- Message Monitoring Panel: recent messages with device ID, timestamp, reason,
  and message content.
- Attack Detection Panel: fake device, replay, flooding, and invalid encrypted
  message events.
- Device Trust Score Panel: simple trust scores calculated from observed log
  behavior.
- Logs Viewer: filterable table of accepted, suspicious, blocked, and attack
  events.

The dashboard does not simulate data. If no logs are present, start Cloud, Fog,
and an IoT or attack client first.
